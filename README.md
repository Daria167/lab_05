
### Мета заняття:
 Розуміти основні концепції систем контролю версій
Створювати та керувати Git репозиторіями
Фіксувати зміни (commits) та працювати з історією
Створювати та об'єднувати гілки (branches)
Працювати з віддаленими репозиторіями (GitHub, GitLab)
Співпрацювати з іншими розробниками через pull requests
Вирішувати конфлікти злиття
Використовувати SSH-ключі та токени безпечно
Працювати з .gitignore та захищати конфіденційні дані

### Контрольні питання: 
 1. Основи Git
Яка різниця між Git та GitHub? Git - це система контролю версій для збереження історії коду. GitHub - це онлайн-сервіс для зберігання репозиторіїв Git і спільної роботи.

Що означає "розподілена" система контролю версій? Кожен користувач має повну копію всього репозиторію з історією змін.

Яка різниця між git init та git clone? Git init - це створює новий локальний репозиторій. Git clone - копіює існуючий репозиторій з сервера.


3. Staging Area та Commits

Що таке Staging Area і навіщо вона потрібна? Це проміжна зона, куди додаються зміни перед комітом.

Яка різниця між git add . та git add -A? Git add - додає конкретний файл Staging Area. Git add -A - додає всі зміни (нові, змінені, видалені) у Staging Area.

Як написати гарне commit message? Використовувати короткий, зрозумілий опис змін у теперішньому часі.

Як виправити останній коміт? git commit --amend - редагує останній коміт.

4. Гілки та злиття
Що таке гілка в Git і чому вони "легковажні"? Це незалежні версії коду; вони легковажні, бо створюються швидко і займають мало місця.
Яка різниця між git merge та git rebase? Git merge об'єднує гілки, зберігаючи історію. Git rebase переносить коміти на нову базу, роблячи історію лінійною.
Що таке fast-forward merge? Це об'єднання гілок без нового коміту, коли історія не розгалужена.
Коли виникають merge конфлікти і як їх уникнути? Конфлікти виникають при змінах одного файлу в різних гілках; уникнути можна частим git pull і маленькими комітами.
Чому не можна робити rebase публічних комітів? Бо це змінює історію, що може зламати роботу інших, хто вже використовує ці коміти.

5. Віддалені репозиторії
Що таке origin? Це стандартне ім'я віддаленого репозиторію, з якого клонувався проєкт.
Яка різниця між git fetch та git pull? Git fetch - завантажує зміни з віддаленого репозиторію; Git pull - завантажує зміни і одразу робить merge або rebase.
Що робить git push --force і чому це небезпечно? Перезаписує історію на сервері; небезпечно, бо може стерти чужі коміти.
Як видалити віддалену гілку? git push origin --delete <назва гілки> - видаляє віддалену гілку.

6. Скасування змін
Яка різниця між git reset --soft, --mixed та --hard? --soft - скасовує коміти, але зберігає зміни у Staging Area. --mixed -скасовує коміти і прибирає зміни зі Staging Area, але лишає у файлах. --hard - повністю скасовує коміти і видаляє всі зміни.
Яка різниця між git reset та git revert? Git reset змінює історію, видаляючи коміти. Git revert створює новий коміт, що скасовує зміни без зміни історії.
Як скасувати git add (видалити файл зі staging)? git reset <файл> - прибирає файл зі Staging Area.
Як повернути видалений файл? git checkout -- <файл> - відновлює файл із останнього коміту.

7. Безпека
Що не повинно потрапляти в Git репозиторій? Паролі, ключі, конфіг-файли, тимчасові файли і системні файли.
Що робити якщо випадково закомітили пароль? Видалити пароль, зробити новий коміт і переписати історію (git rebase), потім змінити пароль 
HTTPS vs SSH для GitHub — що краще? SSH є безпечнішим та зручнішим для постійної роботи; HTTPS простіший для одноразового доступу.
Що таке Personal Access Token і навіщо він потрібен? Ключ для автентифікація в GitHub замість пароль.

8. GitHub
Що таке Pull Request? Це запит на перевірку та об'єднання змін у головну гілку проєкту.
Яка різниця між Issues та Pull Requests? Issues - для обговорення проблем чи ідей. Pull Requests - для пропозиції змін у коді.
Що таке Fork і навіщо він потрібен? Це копія чужого репозиторію у вашому акаунті для власних змін без впливу на оригінал.
Що таке GitHub Actions? Це інструмент для автоматизації процесів, як-от тестування чи деплой коду.

9. Best Practices
Як часто треба робити коміти? Після кожної логічно завершеної зміни в коді.
Скільки часу має жити feature branch? Доки триває робота над однією фічею; бажано кілька днів, не тижнів.
Як організувати git workflow в команді? Використовувати основну гілку(main) і окремі гілки для фіч, робити pull request перед злиттям і код-рев'ю.
Чи треба комітити node_modules або інші залежності? Ні, їх не комітять - замість цього додають у .gitignore.
