# Лабораторна робота №5  
## Система контролю версій Git та співпраця через GitHub

### Мета заняття:
- Розуміти основні концепції систем контролю версій  
- Створювати та керувати Git репозиторіями  
- Фіксувати зміни (commits) та працювати з історією  
- Створювати та об'єднувати гілки (branches)  
- Працювати з віддаленими репозиторіями (GitHub, GitLab)  
- Співпрацювати з іншими розробниками через pull requests  
- Вирішувати конфлікти злиття  
- Використовувати SSH-ключі та токени безпечно  
- Працювати з `.gitignore` та захищати конфіденційні дані  

---

### Контрольні питання

####  Основи Git
**1. Яка різниця між Git та GitHub?**  
Git — це система контролю версій для збереження історії коду.  
GitHub — це онлайн-сервіс для зберігання репозиторіїв Git і спільної роботи.

**2. Що означає "розподілена" система контролю версій?**  
Кожен користувач має повну копію всього репозиторію з історією змін.

**3. Яка різниця між `git init` та `git clone`?**  
`git init` — створює новий локальний репозиторій.  
`git clone` — копіює існуючий репозиторій з сервера.

---

####  Staging Area та Commits
**1. Що таке Staging Area і навіщо вона потрібна?**  
Це проміжна зона, куди додаються зміни перед комітом.

**2. Яка різниця між `git add .` та `git add -A`?**  
`git add <файл>` — додає конкретний файл у Staging Area.  
`git add -A` — додає всі зміни (нові, змінені, видалені).

**3. Як написати гарне повідомлення до коміту (commit message)?**  
Використовуй короткий, зрозумілий опис змін у теперішньому часі.

**4. Як виправити останній коміт?**  
`git commit --amend` — редагує останній коміт.

---

####  Гілки та злиття
**1. Що таке гілка в Git і чому вони "легковажні"?**  
Це незалежні версії коду; вони легковажні, бо створюються швидко і займають мало місця.

**2. Яка різниця між `git merge` та `git rebase`?**  
`git merge` об’єднує гілки, зберігаючи історію.  
`git rebase` переносить коміти на нову базу, роблячи історію лінійною.

**3. Що таке fast-forward merge?**  
Це об’єднання гілок без нового коміту, коли історія не розгалужена.

**4. Коли виникають merge-конфлікти і як їх уникнути?**  
Конфлікти виникають при змінах одного файлу в різних гілках.  
Уникнути можна частим `git pull` і малими комітами.

**5. Чому не можна робити rebase публічних комітів?**  
Бо це змінює історію, що може зламати роботу інших розробників.

---

####  Віддалені репозиторії
**1. Що таке origin?**  
Це стандартне ім’я віддаленого репозиторію, з якого клонувався проєкт.

**2. Яка різниця між `git fetch` та `git pull`?**  
`git fetch` — завантажує зміни без злиття.  
`git pull` — завантажує зміни та одразу виконує merge або rebase.

**3. Що робить `git push --force` і чому це небезпечно?**  
Перезаписує історію на сервері; небезпечно, бо може стерти чужі коміти.

**4. Як видалити віддалену гілку?**  
`git push origin --delete <назва_гілки>`

---

####  Скасування змін
**1. Яка різниця між `git reset --soft`, `--mixed` та `--hard`?**  
- `--soft` — скасовує коміти, але зберігає зміни у Staging Area.  
- `--mixed` — скасовує коміти та прибирає зміни зі Staging Area, але лишає у файлах.  
- `--hard` — повністю видаляє зміни.

**2. Яка різниця між `git reset` та `git revert`?**  
`git reset` змінює історію, видаляючи коміти.  
`git revert` створює новий коміт, що скасовує зміни без зміни історії.

**3. Як скасувати `git add` (видалити файл зі Staging Area)?**  
`git reset <файл>`

**4. Як повернути видалений файл?**  
`git checkout -- <файл>`

---

####  Безпека
**1. Що не повинно потрапляти в Git репозиторій?**  
Паролі, ключі, конфігураційні та тимчасові файли.

**2. Що робити, якщо випадково закомітили пароль?**  
Видалити пароль, зробити новий коміт, переписати історію (`git rebase`) і змінити пароль.

**3. HTTPS vs SSH для GitHub — що краще?**  
SSH є безпечнішим і зручнішим для постійної роботи; HTTPS — простіший для одноразового доступу.

**4. Що таке Personal Access Token і навіщо він потрібен?**  
Це ключ для автентифікації в GitHub замість пароля.

---

####  GitHub
**1. Що таке Pull Request?**  
Запит на перевірку та об’єднання змін у головну гілку проєкту.

**2. Яка різниця між Issues та Pull Requests?**  
Issues — для обговорення проблем чи ідей.  
Pull Requests — для пропозиції змін у коді.

**3. Що таке Fork і навіщо він потрібен?**  
Це копія чужого репозиторію у вашому акаунті для власних змін без впливу на оригінал.

**4. Що таке GitHub Actions?**  
Інструмент для автоматизації процесів, як-от тестування чи деплой коду.

---

####  Best Practices
**1. Як часто треба робити коміти?**  
Після кожної логічно завершеної зміни в коді.

**2. Скільки часу має жити feature branch?**  
Доки триває робота над однією фічею — бажано кілька днів, не тижнів.

**3. Як організувати git workflow у команді?**  
Використовувати основну гілку `main` і окремі гілки для фіч, робити Pull Request перед злиттям та код-рев’ю.

**4. Чи треба комітити `node_modules` або інші залежності?**  
Ні. Їх додають у `.gitignore`.
